## 1. Executive Summary

This document establishes the precise, measurable, and programmatic criteria required for a component, feature, or subsystem to be marked as **Done** inside the BankYar project. No code may be merged, released, or reviewed unless it meets 100% of these quality gates.

---

## 2. Purpose & Scope

### Purpose
To eliminate architectural drift, linter decay, and regression issues by enforcing clear acceptance and quality criteria on every file modified or created.

### Scope
Applies to all pull requests, feature branches, and code generated by humans or AI agents.

---

## 3. Definitions

| Term | Definition |
| :--- | :--- |
| **Quality Gate** | A set of automated or static requirements that code must pass before moving to the next pipeline stage. |
| **Linter Hint** | A style or minor optimization suggestion flagged by the static analyzer. |

## 4. Measurable Definition of Done (DoD)

A development task is considered **Done** only when it fulfills all criteria within the following checklist categories:

```
       +--------------------------------------------------------+
       |               DEFINITION OF DONE CHECKS                |
       +--------------------------------------------------------+
       |   CRITERIA            |      MEASURABLE TARGET         |
       +-----------------------+--------------------------------+
       |  - Compilation        |  - 0 Errors, 0 Warnings        |
       |  - File Size          |  - Under 300 lines of code     |
       |  - Class Size         |  - Under 250 lines of code     |
       |  - Method Size        |  - Under 30 lines of code      |
       |  - Test Coverage      |  - 85% base, 100% for parsers  |
       |  - Documentation      |  - 100% public APIs with ///   |
       |  - Network Protection |  - 0 Internet permissions      |
       +-----------------------+--------------------------------+
```

### 4.1 Static Analysis & Compilation
- [ ] Code compiles with exactly **0 errors**, **0 warnings**, and **0 linter hints** on all target platform environments.
- [ ] No file exceeds the absolute **300-line** cap.
- [ ] No class contains more than **250 lines** or exposes more than **5 public methods**.
- [ ] All public methods are kept under **30 lines** with a cyclomatic complexity strictly below **8**.
- [ ] The official Dart formatter is executed across all modified files (`dart format --line-length=80 .`).

### 4.2 Clean Architecture compliance
- [ ] The `domain/` directory contains 100% pure Dart. There are absolutely no imports from Flutter visual libraries, Riverpod, SQLCipher, or platform integrations.
- [ ] Database DAOs and raw SQL queries are completely hidden inside repositories and data sources; no UI file has direct access to persistence classes.
- [ ] Dependencies point exclusively inward towards the core domain. Circular dependencies are strictly avoided.

### 4.3 Security & Privacy Protection
- [ ] The main `AndroidManifest.xml` has been verified to contain exactly **zero** internet permissions (`android.permission.INTERNET` is absent).
- [ ] Sensitive screen layouts (containing transactions, balances, or credentials) implement secure overlay protection (`FLAG_SECURE` or platform-native blurs) to redact details from multitasking views and screenshots.
- [ ] No database decryption keys or credentials exist as plaintext or hardcoded values inside files.

### 4.4 Accessibility (a11y) & Localization (l10n)
- [ ] Visual layouts have been tested to scale dynamically up to **200% font size** without throwing layout errors or causing text overlap.
- [ ] All interactive components have clear, descriptive screen reader labels using Flutter `Semantics`.
- [ ] Right-to-Left (RTL) layout mirroring works dynamically for Persian users, including mirrored arrows, lists, and sliders.
- [ ] Zero hardcoded string labels are present; all text is loaded dynamically via local Farsi and English `l10n` templates.

### 4.5 Automated Testing Gates
- [ ] Unit test coverage on the affected code meets or exceeds **85%** (**100% coverage** is mandatory for all core SMS parser regular expressions).
- [ ] All unit, widget, and integration tests execute and pass without failure.
- [ ] Test execution contains exactly zero arbitrary delays (`sleep`); mock clocks and schedulers are used instead.

## 5. Best Practices & Examples

### Best Practices
- **Verify early:** Do not wait until raising a Pull Request to run static analysis. Run `dart analyze` after every code save.
- **Fail the Build on Warns:** Configure your editor to highlight linter warnings as blocking errors.

### Example DoD Compliance Checklist Run
```
[Pass] Compilation verified (0 warnings).
[Pass] File size: lib/features/notes/domain/entities/note_entity.dart is 45 lines.
[Pass] Clean Domain check: No material imports present in domain.
```

## 6. Common Mistakes & Anti-patterns

- **The "Almost Done" PR:** Requesting a review on code that fails static analysis under the assumption that "minor linter hints can be resolved later."
- **Bypassing the Formatter:** Committing code formatted by non-standard editors, leading to formatting discrepancies.

---

## 7. Recommendations & Implementation Notes

- **Enforce compiler options:** Ensure that the static compiler remains configured with zero exceptions inside `analysis_options.yaml`.
- **Pre-Commit Formatter Automation:** Set up IDE "Format on Save" to ensure 100% adherence to standard spacing rules.

## 8. Layer-Specific DoD Requirements

### 8.1 UI Widgets & Screens DoD
- [ ] Widget build methods are under **40 lines of code**.
- [ ] Visual nesting tree does not exceed **3 levels deep**.
- [ ] State is accessed exclusively via unidirectional Riverpod providers.

### 8.2 UseCases DoD
- [ ] Implements a single business action with exactly one public executing method (`call` or `execute`).
- [ ] Purely stateless; dependencies are final and injected via constructor.
- [ ] Returns failures safely using functional union or `Result` types instead of throwing unhandled exceptions.

### 8.3 Repositories DoD
- [ ] Concrete implementations capture platform and SQL cipher errors and map them to standard Domain `Failure` types before returning.
- [ ] Restricts public method count to **5 or fewer**.

---

## 9. Review Checklist

Verify the following before final merge:
- [ ] The feature is completely self-contained under `lib/features/[feature_name]/`.
- [ ] Run validator tools (`layout_validator.py`, `accessibility_validator.py`, etc.) and ensure 100% compliance.
- [ ] Run the complete test suite (`flutter test`) and confirm all pass.
