## 1. AI Engineering Philosophy & Governance

### 1.1 AI Engineering Philosophy
The BankYar engineering ecosystem is developed almost entirely through AI-assisted and AI-driven engineering pipelines. Our philosophy regarding Artificial Intelligence in software engineering is defined by three core pillars:

1. **AI as a First-Class Engineer, Human as the Governor:** AI is not a mere autocomplete utility; it is a collaborative, high-capability execution agent. Humans act as orchestrators, policy enforcers, and validators.
2. **Deterministic Governance over Stochastic Systems:** Large Language Models (LLMs) are probabilistic and stochastic. To guarantee enterprise-grade safety, security, and architectural correctness, their outputs must be constrained, validated, and evaluated using rigorous, deterministic workflows and strict quality gates.
3. **Zero-Trust AI Output:** All code, architectures, and tests generated by AI are assumed to be flawed, incomplete, or divergent until proven correct through rigorous automated static analysis, comprehensive compilation verification, and self-review protocols.

### 1.2 AI Governance Principles
To ensure long-term maintainability, the following principles govern all AI activities:

* **Principle of Least Context:** Do not flood the AI with irrelevant files. Provide the exact, high-density context required for the current scope to maximize focus and minimize context degradation or hallucinations.
* **Architecture-First Bound:** AI models must operate under the absolute constraint of existing architectural specifications (`ARCHITECTURE.md`, `CODING_CONSTITUTION.md`, `SECURITY_ARCHITECTURE.md`, etc.). The AI has no authority to alter base architectural paradigms unless explicitly instructed in a dedicated architectural revision epoch.
* **Verifiable Correctness:** Every structural generation must produce code that is testable in isolation. If a generated component cannot be unit tested without extensive external infrastructure, the generation is rejected.
* **No Code Leaks:** AI must keep all financial, domain, and security rules strictly local. No remote service integrations, cloud analytics, or external telemetry hooks can be introduced by the AI.

## 3. Prompt Engineering Standards & Structure

Every developer prompt sent to an AI assistant within the BankYar workspace must be structured according to our high-density system. This maximizes predictability and prevents context drift.

### 3.1 Prompt Structure Specification
A standard BankYar engineering prompt consists of six distinct blocks, ordered as follows:

```
+-----------------------------------------------------------------------+
| 1. CORE ROLE & SYSTEM BEHAVIOR (Establishes specialized agent domain) |
+-----------------------------------------------------------------------+
| 2. ARCHITECTURAL BOUNDARY REFERENCES (Links to coding standards)      |
+-----------------------------------------------------------------------+
| 3. TARGET ENVIRONMENT & TECHNICAL STACK (Framework, state, database)  |
+-----------------------------------------------------------------------+
| 4. DESIRED OBJECTIVE & FUNCTIONAL REQUIREMENTS (The exact task scope) |
+-----------------------------------------------------------------------+
| 5. INPUT CONTEXT & FILE MAP (Strict paths of files to read/edit)      |
+-----------------------------------------------------------------------+
| 6. EXPECTED FORMAT & QUALITY CONSTRAINTS (Lines, rules, output format)|
+-----------------------------------------------------------------------+
```

### 3.2 Context Management & Window Policy
* **Context Budgeting:** The workspace limits active context injection to **100,000 tokens** per session to avoid the "needle-in-a-haystack" degradation phenomenon.
* **Precise File Targeting:** Never ingest the whole repository. The prompt author must provide a precise list of active files (maximum 5) and auxiliary read-only files (maximum 5).
* **Reference Anchoring:** All files passed to the context window must begin with a clear file path comment banner (e.g., `// File: lib/features/transactions/domain/entities/transaction_entity.dart`).

## 4. Workflows & Lifecycle Governance

### 4.1 Epic → Story → Task → Prompt Workflow
To avoid architectural drift, feature requests are decomposed incrementally down to atomic, single-responsibility prompt executions.

```
+------------------+     Decomposed     +--------------------+
|  EPIC BACKLOG    | -----------------> | USER STORY DECK    |
| (e.g. Analytics) |                    | (e.g. Chart View)  |
+------------------+                    +---------+----------+
                                                  |
                                                  | Structured Split
                                                  v
+------------------+      Mapped to     +--------------------+
| SINGLE PROMPT    | <----------------- | ATOMIC TASK        |
| (e.g. Model DTO) |                    | (e.g. Chart DTO)   |
+------------------+                    +--------------------+
```

### 4.2 Feature Development Workflow Decision Table

| Step | Action | AI Responsibility | Human Responsibility | Validation Gate |
| :--- | :--- | :--- | :--- | :--- |
| **1. Planning** | Context Analysis & Plan Generation | Analyze feature scope and propose a step-by-step plan. | Review the proposed plan; approve or request revisions. | User Approval of Plan |
| **2. Scaffolding** | Folder & Abstract Interface Definition | Generate folder structure and abstract domain contracts. | Verify path alignments and contract logic. | 0 Static Analysis Errors |
| **3. Core Logic** | Data Models & Domain Use Cases | Generate immutable entities, DTOs, and pure Dart use cases. | Perform code review on layer purity. | No external imports in Domain |
| **4. Presentation** | State Notifiers & Screen Widgets | Generate Riverpod StateNotifiers and declarative widgets. | Review UI rendering complexity and limits. | Build method < 40 lines |
| **5. Verification** | Unit, Integration, and UI Tests | Generate exhaustive unit tests and widget tests. | Execute tests locally. | > 85% Test Coverage |

### 4.3 Refactoring Workflow
When refactoring existing components, the AI must strictly adhere to the following sequence:
1. **Dependency Analysis:** Generate a dependency map of the target file to determine affected modules.
2. **Behavioral Baseline Safeguard:** Before applying code changes, the AI must verify that existing unit tests pass.
3. **Targeted Extraction:** Refactor components by breaking down classes that exceed **250 lines** or functions that exceed **30 lines** into single-purpose components.
4. **Incremental Compilation Verification:** Run the Dart compiler after each atomic component refactor.

### 4.4 Bug Fix Workflow
The AI must follow a structured root-cause analysis cycle:

```
           +---------------------------------------------+
           |           1. DEFINE SPECIFIC FAILURE        |
           |   - Capture full stack trace & error logs   |
           +----------------------+----------------------+
                                  |
                                  v
           +---------------------------------------------+
           |          2. ISOLATE CRITICAL BOUNDARY       |
           |   - Identify the exact layer (Data/Domain)  |
           +----------------------+----------------------+
                                  |
                                  v
           +---------------------------------------------+
           |       3. REPLICATE FAILURE VIA TEST         |
           |   - Write a failing unit/integration test   |
           +----------------------+----------------------+
                                  |
                                  v
           +---------------------------------------------+
           |            4. IMPLEMENT BUG REPAIR          |
           |   - Apply fix; verify failing test passes   |
           +----------------------+----------------------+
                                  |
                                  v
           +---------------------------------------------+
           |         5. REGRESSION CHECK & CLEANUP       |
           |   - Execute entire feature test suite       |
           +---------------------------------------------+
```

### 4.5 Self-Review & Architecture Validation Workflow
Prior to outputting code, the AI is programmatically required to evaluate its output against the active architectural constraints. It must run a cognitive validation loop answering:
* *Did I place any UI elements or presentation packages (e.g. Flutter Material) into the `domain/` directory?*
* *Are all properties in my generated Domain classes marked `final` to ensure immutability?*
* *Did I directly invoke database layers inside state holders instead of going through abstract repository contracts?*
* *Does this generation violate the maximum file-size limit (300 lines) or cyclomatic complexity (8)?*

## 7. Prevention and Mitigation Strategies

LLM tools, although highly productive, carry several technical risks. We enforce three core mitigation strategies to maintain absolute compliance with BankYar principles.

### 7.1 Hallucination Prevention Strategy
* **Zero Speculative Generation:** The AI is strictly forbidden from writing code for hypothetical future extensions unless a formal feature schema and interface are requested in the prompt.
* **Grounded Source Compilation:** If the AI needs to use API parameters, it must first inspect the library definition file (e.g., examining the package version inside `pubspec.yaml` and checking public class parameters) rather than guessing signatures.
* **Failure to Generate Protocol:** If an AI model cannot find a deterministic path to fulfill a task because of missing contracts or conflicting rules, it must halt and request human clarification via `request_user_input` rather than generating a best-guess approximation.

### 7.2 Duplicate Code Prevention Strategy
* **Pre-generation Search Audit:** Before generating any parsing, conversion, or formatting logic, the AI must search `lib/core/` and `lib/features/` to find existing utility methods.
* **Extraction Over Duplication:** If identical logic is required in two features, the AI must extract that routine into a static pure function inside a shared utility file in `lib/core/` and import it across both modules.

### 7.3 Context Drift & Decay Prevention
* **Short-Session Ephemerality:** AI conversations and context windows must be periodically flushed (e.g. starting a fresh chat session every **20 interactions** or after completing an atomic task) to remove historical conversational noise.
* **Incremental Checkpointing:** After completing a task, the AI must summarize the changes made and write them to a localized developer memory file, allowing subsequent sessions to initialize with a clean, low-token context outline.

This Constitution is not a static document. As foundation models evolve and the BankYar ecosystem expands, the governance framework will adapt:
1. **Technical Debt Reviews:** Every release cycle dedicates **20%** of human-AI collaboration bandwidth to resolving lint warnings, refactoring complex files, and updating prompt templates.
2. **On-Device ML Evolutions:** While Phase 1 is strictly deterministic and regex-based, the prompt engineering templates are prepared to cleanly integrate lightweight on-device models (e.g., TensorFlow Lite) within the data datasource layer without affecting the Domain or Presentation.
3. **No-Network Immutable Lock:** Under no circumstances can any future update, configuration, or continuous improvement process introduce a network permission to BankYar. The application must remain an offline-only, local sanctuary for user financial data.
